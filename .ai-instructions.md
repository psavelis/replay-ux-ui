# AI Assistant Instructions for leetgaming-pro-web

## Terminal Command Best Practices

### ❌ DON'T: Use problematic patterns
```bash
# BAD - Can cause hangs if command fails
command 2>&1 | head -50

# BAD - Can hang on interactive prompts
npm install
yarn install

# BAD - Long-running processes without background flag
npm run dev
npm start
```

### ✅ DO: Use safe patterns
```bash
# GOOD - Check exit code first
command; echo "Exit code: $?"

# GOOD - Use grep/awk instead of head for filtering
command 2>&1 | grep -A 5 "error"

# GOOD - Limit output at source when possible
command --max-results 50

# GOOD - Use timeout for potentially long commands
timeout 10s command

# GOOD - Check if command exists first
which tsc && tsc --version || echo "tsc not found"
```

## TypeScript Compilation Checks

```bash
# Check if tsc exists and run
cd /path/to/project && npx tsc --version

# Get errors only (safer than full compilation)
cd /path/to/project && npx tsc --noEmit 2>&1 | grep -E "error TS[0-9]+" | head -20

# Alternative: Use the errors tool instead
# Use get_errors tool with file paths
```

## File Operations

### Prefer built-in tools over shell commands
- Use `read_file` instead of `cat`
- Use `list_dir` instead of `ls`
- Use `file_search` instead of `find`
- Use `grep_search` instead of `grep -r`

## Build/Install Operations

### Never run these without explicit user request:
- `npm install` / `yarn install` / `pnpm install`
- `npm run build` / `yarn build`
- `npm run dev` / `yarn dev`
- Any Docker commands that start containers

### If you need to check installations:
```bash
# Check package.json for dependencies
grep "dependency-name" package.json

# Check if package is installed
ls node_modules/package-name 2>/dev/null && echo "installed" || echo "not installed"

# Check global installations
npm list -g --depth=0 | grep package-name
```

## Error Handling

1. **Always use the `get_errors` tool first** before attempting terminal compilation checks
2. If `get_errors` is not sufficient, ask the user before running compilation
3. For build errors, focus on specific files rather than full project compilation

## Code Modifications

1. **DRY Principle**: Always check existing code before creating new files
2. **SOLID Principles**: Maintain single responsibility and dependency inversion
3. **Incremental Changes**: Make small, testable changes
4. **Type Safety**: Ensure all TypeScript types are properly defined

## Documentation

- **Do NOT** create summary markdown files unless explicitly requested
- **Do** update README.md when adding major features
- **Do** add inline comments for complex logic
- **Do** create examples.ts for usage patterns

## Testing

Before marking work complete:
1. Use `get_errors` to check for compilation errors
2. Verify imports are correct
3. Check that all required dependencies exist
4. Ensure no breaking changes to existing code

## This Project Specific Notes

- **Mock Replacement**: Always use real SDK implementations, not mocks
- **Authentication**: RID tokens managed by `RIDTokenManager` singleton
- **Resource Ownership**: All entities have hierarchical ownership model
- **API Client**: Use `ReplayApiClient` or `UploadClient`, not raw fetch/axios
